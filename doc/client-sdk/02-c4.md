# 容器/组件级视图与「未来态」

采用 Mermaid 表达 **C4 风格** 的容器与组件：实线=当前主路径，虚线=规划中的扩展。非严格 C4-PlantUML 语法，但便于在 Markdown 中维护。

---

## 1. 容器图（C4-Container 风格）

```mermaid
flowchart TB
    subgraph app["业务进程"]
        U["业务代码"]
    end

    subgraph rflow["librflow_client 动态库/静态库"]
        API["C API<br/>lifecycle / stream / …"]
        ST["rflow::client::State"]
        MGR["WebRtcPullManager"]
        WPS["WebRtcPullStream"]
        SC["impl/SignalingClient<br/>per-stream TCP+JSON"]
    end

    subgraph core["librflow core"]
        RT["rflow::rtc<br/>Factory + 线程"]
        TH["rflow::thread<br/>线程池（当前少用到 Client）"]
        SIG["rflow::signal<br/>当前 stub"]
    end

    RMT["对端/媒体"]
    SV["信令服务"]

    U -->|librflow_*| API
    API --> ST
    API --> MGR
    MGR --> WPS
    WPS --> SC
    WPS --> RT
    MGR --> RT
    SC -->|TCP/JSON| SV
    WPS -->|RTP/ICE| RMT
    API --> TH
    API --> SIG

    style SIG stroke-dasharray: 5 5
    style TH stroke-dasharray: 3 3
```

说明：

- **当前**：开流、SDP/ICE 由 `WebRtcPullStream` + 每路 `SignalingClient` 完成；`rflow::rtc` 为硬依赖；`rflow::thread` / `rflow::signal` 在 `init` 中拉起，Client 主路径几乎未用。
- **虚线意涵**：`thread` 用于将来统一回调派发；`signal` 为将来设备级长连信令抽象位。

## 2. 组件图 + 未来态（虚线）

```mermaid
flowchart LR
    subgraph now["当前"]
        A[librflow_client_api.h]
        B[lifecycle + stream + infrastructure]
        C[WebRtcPullManager]
        D[WebRtcPullStream]
        E[SignalingClient]
        F[rflow::rtc]
    end

    A --> B
    B --> C
    C --> D
    D --> E
    D --> F

    subgraph future["规划（非当前实现）"]
        G[core/signal 长连<br/>WS/自研/直连]
        H[rflow::thread 池<br/>统一 dispatch]
    end

    B -.->|将来: 设备鉴权/notice| G
    D -.->|将来: 回调不直连业务线程| H
    E -.->|将来: 可废弃或降级为| G

    style G stroke-dasharray: 5 5
    style H stroke-dasharray: 5 5
```

**评审时可口述**：`connect` 若接入真实设备握手，可落在 `B` 与 `G` 之间；每流 TCP 可收敛为**单连接多路**或**共享 G 的会话**（需协议升级）。

## 3. 与公共头文件的关系

| 头文件 / 配置 | 作用 |
|---------------|------|
| `librflow_client_api.h` | 对外 C ABI；业务只依赖于此 |
| `librflow_common.h` | 日志、global_config、video_frame 等 |
| `librflow_global_config` 中 `signal` | 当前用于 **URL 等** 注入；与 `rflow::signal` 实现层尚未合一 |
